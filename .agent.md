# Skill Evolution Agent

**Purpose:** Specialized agent customization for evolving SKILL.md definitions, test cases, and evaluator rules during distillation (or standalone skill optimization).

---

## When to Use This Agent

✅ **Use this agent for:**
- Adding or modifying SKILL.md definitions
- Creating or refining test cases and fixtures
- Improving rule-based evaluator checks
- Analyzing skill portability across models
- Debugging agent tool usage patterns

❌ **Do NOT use for:**
- General Python development (use default agent)
- Infrastructure changes (use main agent)
- Non-skill-related tasks

---

## Skill-Specific Conventions

### SKILL.md Structure

Every SKILL.md must have:

```yaml
---
name: docx
version: 1.0
description: >
  Create, read, edit, and convert Word (.docx) documents.
  Supports formatting, tables, images, tracked changes.
last_updated: 2026-04-15
models_tested:
  - qwen/qwen3-8b (score: 0.72)
  - qwen/qwen3-14b (score: 0.81)
  - anthropic/claude-haiku-4-5 (baseline: 0.95)
---

# Skill: Create and Edit DOCX Files

## Overview
[clear, concise description of what the skill does]

## Tools Available
[list tools with their purpose and constraints]

## Before You Start
[prerequisites, important notes, common pitfalls]

## Step-by-Step Examples
[3-5 concrete examples showing tool usage]

## Common Tasks
[troubleshooting, edge cases, recovery strategies]
```

**Key Rules:**
- Max 2000 chars per SKILL.md (keep it concise)
- Examples must be complete and runnable
- Avoid jargon; assume reader is unfamiliar with tool
- Explicitly list which workflows it supports (Create/Read/Edit/Convert/Edge)
- Include edge case handling (corrupted files, timeouts, unsupported formats)

### Test Case Schema (docx.json for reference)

```json
{
  "test_case_id": "tc_<workflow><number>",
  "workflow": "A || B || C || D || E",
  "prompt": "Human-readable task description",
  "description": "Why this test case matters",
  "rule_checks": [
    {
      "type": "xml.contains",
      "xpath": "//w:body/w:p/w:r/w:rPr/w:b",
      "description": "Document must have bold-formatted text"
    },
    {
      "type": "file.exists",
      "path": "output.docx",
      "description": "Agent must create output.docx file"
    }
  ],
  "content_checks": [
    {
      "keywords": ["Report", "Q1"],
      "description": "Output text must mention Report and Q1"
    }
  ],
  "must_have_docx": true,
  "fixtures": ["sample_report.docx"],
  "estimated_difficulty": "medium"
}
```

**Workflows:**
- `A`: Create new document
- `B`: Read/extract from existing document
- `C`: Edit existing document (in-place modification)
- `D`: Convert between formats (docx → pdf, etc.)
- `E`: Edge cases (corrupted files, limits, error handling)

### Evaluator Rule Checks (docx_rules.py template)

```python
class DocxRuleEvaluator(RuleEvaluator):
    """Evaluates DOCX file outputs against rule checks."""

    def __init__(self, output_dir: str):
        self.file_path = os.path.join(output_dir, "output.docx")

    def evaluate(self, test_case: dict) -> EvalResult:
        """Run all checks for this test case."""
        checks = []

        # Check 1: File exists
        file_check = self._check_file_exists()
        checks.append(file_check)

        if not file_check.passed:
            # Skip content checks if file missing
            return EvalResult(
                test_case_id=test_case["test_case_id"],
                rule_checks=checks,
                rule_score=0.0,
                passed=False
            )

        # Check 2: XML validity
        xml_check = self._check_xml_validity()
        checks.append(xml_check)

        # Check 3: Specific style/content rules
        style_checks = self._check_required_styles(
            test_case.get("rule_checks", [])
        )
        checks.extend(style_checks)

        # Score: average of all checks
        rule_score = sum(c.score for c in checks) / len(checks) if checks else 0

        return EvalResult(
            test_case_id=test_case["test_case_id"],
            rule_checks=checks,
            rule_score=rule_score,
            passed=rule_score >= 0.6
        )

    def _check_file_exists(self) -> CheckResult:
        """Verify output.docx exists."""
        ...

    def _check_xml_validity(self) -> CheckResult:
        """Parse and validate DOCX internal XML."""
        ...

    def _check_required_styles(self, rule_checks) -> list[CheckResult]:
        """Apply user-defined rule checks (XPath, style properties)."""
        ...
```

## Development Workflows

### Workflow 1: Optimizing an Existing SKILL.md

**Goal:** Improve a SKILL.md for better small-model performance

**Steps:**
1. Read current SKILL.md from `skill_runner/skills/<skill>/SKILL.md`
2. Run distillation: `/create-distillation-task` prompt
3. Analyze failures: `distillation-debugger` agent
4. Suggest changes to SKILL.md (based on teacher's key_notes + debug analysis)
5. Update version number and last_updated timestamp
6. Document changes in commit message

**Example:**
```
The teacher noted that tc_a05 fails because SKILL.md doesn't explain
nested XML formatting. I'll add explicit examples:

OLD:
  "Apply formatting as needed"

NEW:
  "For multiple formatting (bold + italic), nest tags:
   <w:r><w:rPr><w:b/><w:i/></w:rPr><w:t>Bold Italic</w:t></w:r>"
```

### Workflow 2: Creating a New Test Case

**Goal:** Add a test case that exposes a specific skill gap

**Steps:**
1. Identify gap (from distillation failures or model benchmark)
2. Create fixture file (if input needed):
   - Place in `distillation/test_cases/fixtures/<name>.ext`
   - Keep small (~50 KB max to avoid API costs)
3. Write test case to `distillation/test_cases/<skill>.json`:
   - ID: `tc_<workflow><next_number>` (e.g., tc_a32 if last was a31)
   - Write clear prompt that exposes the gap
   - Define rule_checks (what the output MUST have)
   - Define content_checks (semantic validation)
4. Run test case on current model: `skill_runner`: `python main.py run --skill docx --prompt "..." --verbose`
5. Verify it's challenging (score < 0.8 on baseline)

**Bad test case (too easy):**
```json
{
  "prompt": "Create a document",
  "rule_checks": []  // Empty; everything passes
}
```

**Good test case (clear requirements):**
```json
{
  "prompt": "Create a report with a title in bold, a table with 3 columns, and a bulleted list",
  "rule_checks": [
    {"type": "xml.contains", "xpath": "//w:r/w:rPr/w:b", "description": "Must have bold text"},
    {"type": "xml.contains", "xpath": "//w:tbl", "description": "Must have a table"},
    {"type": "xml.contains", "xpath": "//w:pPr/w:numPr", "description": "Must have bullet list"}
  ],
  "content_checks": [
    {"keywords": ["Report"], "description": "Output must mention Report"}
  ]
}
```

### Workflow 3: Adding a New Skill

**Goal:** Create a new skill (e.g., `xlsx` for Excel) from scratch

**Steps:**
1. Create folder: `skill_runner/skills/xlsx/`
2. Create `SKILL.md` with:
   - Clear description of what the skill does
   - Tools available (write_file, read_file, bash for LibreOffice, etc.)
   - 3-5 concrete examples
3. Create `scripts/` (optional) for helper scripts
4. Create evaluator: `distillation/evaluator/xlsx_rules.py`
   - Inherit from `RuleEvaluator`
   - Implement checks for spreadsheet outputs (cell values, formulas, styles)
5. Register evaluator in `distillation/orchestrator.py`:
   ```python
   from distillation.evaluator.xlsx_rules import XlsxRuleEvaluator

   EVALUATORS = {
       "docx": DocxRuleEvaluator,
       "xlsx": XlsxRuleEvaluator,  # Add this
   }
   ```
6. Create test cases: `distillation/test_cases/xlsx.json` (20-30 cases)
7. Run pilot distillation: `python run.py --skill xlsx --rounds 1 --test-cases 5`

## Code Style & Patterns

### SKILL.md Writing

**Good:**
```markdown
## Examples

### Create a simple report
1. Use write_file to create output.docx
2. Add a heading: <w:p><w:pPr><w:pStyle val="Heading1"/></w:pPr><w:r><w:t>Title</w:t></w:r></w:p>
3. Add a paragraph: <w:p><w:r><w:t>Body text</w:t></w:r></w:p>
```

**Avoid:**
```markdown
## Create Documents
Create document files using XML. The system handles everything automatically.
(Vague; no concrete steps; assumes prior knowledge)
```

### Evaluator Rules

**Good:**
```python
def _check_formula_present(self) -> CheckResult:
    """Verify spreadsheet contains at least one formula (SUM, AVG, etc)."""
    try:
        wb = openpyxl.load_workbook(self.file_path, data_only=False)
        has_formula = any(
            cell.value and isinstance(cell.value, str) and cell.value.startswith("=")
            for ws in wb.worksheets
            for cell in ws.iter_rows()
        )
        return CheckResult(
            name="formulas_present",
            passed=has_formula,
            score=1.0 if has_formula else 0.0,
            reason="Spreadsheet contains formulas" if has_formula else "No formulas found"
        )
    except Exception as e:
        return CheckResult(name="formulas_present", passed=False, score=0.0, reason=str(e))
```

**Avoid:**
```python
def check_spreadsheet(self):
    # TODO: implement
    return True  # Incomplete; no error handling
```

## Quality Checklist

Before committing SKILL.md or test case changes:

- [ ] SKILL.md is ≤2000 chars (concise, focused)
- [ ] Examples are complete & executable (test manually first)
- [ ] Test cases have clear rule_checks (not ambiguous)
- [ ] Test cases have realistic difficulty (mix of easy/medium/hard)
- [ ] Evaluator rules are idempotent (same input → same score every time)
- [ ] Edge cases are documented (corrupted files, timeouts, missing tools)
- [ ] version number bumped in SKILL.md frontmatter
- [ ] last_updated date is today
- [ ] All fixtures are tracked (not in .gitignore)
- [ ] Models tested list is accurate
